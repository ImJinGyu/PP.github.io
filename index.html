Notion이라는 완성된 상업용 SaaS가 있음에도 직접 만들어보고 사용해보고 싶었습니다. 새로운 기술 스택을 실제 프로젝트에 적용해보고 싶었고, Claude Code 기반의 AI 협업 개발이 실무에서 얼마나 유효한지 직접 검증해보고 싶었습니다. 학습 목적과 포트폴리오, 개인 사용 목적의 서비스 구축을 동시에 목표로 했습니다. 실제로 동작하는 서비스로 구축했지만 상업적 운영보다는 기술 학습과 경험에 초점을 둔 프로젝트입니다. AI 협업 개발 방식과 새로운 생태계를 직접 경험해보며 기존 업무에서도 활용할 수 있는 시야를 넓히고 싶었습니다.

새로운 기술 스택 경험

기존에 익숙했던 .NET 환경에서 벗어나 Next.js, Supabase, Vercel을 직접 써보고 싶었습니다. 데이터베이스는 직접 서버를 운영하는 대신 Supabase를 활용했고, 배포도 Vercel을 통해 자동화할 수 있었습니다. 여기서 더 나아가 GitHub Actions로 CI 파이프라인을 직접 구성해, main 브랜치에 push할 때마다 자동으로 빌드 검증이 실행되도록 했습니다. 낯선 환경이었지만 클라이언트-서버 흐름부터 CI/CD 파이프라인까지 전체 구조를 직접 구축하며 빠르게 파악할 수 있었습니다.

AI와 함께 개발한다는 것

코드를 직접 작성하는 대신 요구사항을 문서로 정리해서 AI에 전달하는 방식으로 개발했습니다. CLAUDE.md 파일에 기술 스택, 페이지별 기능 명세, UI/UX 동작 방식, 디자인 방향, 산출물 목록까지 항목별로 구체적으로 정의한 뒤 Claude Code에 전달하고 구현을 맡겼습니다. 결국 요구사항을 얼마나 잘 정의하느냐와 디버깅 능력이 AI 협업 개발의 핵심임을 체감했습니다.

AI와 협업하며 겪은 시행착오

AI의 한계도 직접 체감했습니다. Claude Code는 컨텍스트 한계로 인해 대화가 길어질수록 이전 작업 내용을 잊어버리는 문제가 있었는데, 초반에 맞춰뒀던 디자인 톤이나 컴포넌트 구조를 AI가 기억하지 못하고 임의로 바꾸거나, 새로운 기능을 추가하는 과정에서 기존 기능이 의도치 않게 롤백되는 현상도 발생했습니다. 이를 보완하기 위해 작업 단위마다 AI에게 자동 품질 검사 및 셀프체크 리마인더 등 스스로 검토하고 변경사항을 체크리스트와 변경 로그로 기록하게 하는 작업 방식을 만들어 컨텍스트 한계에 도달했을 때 주기적으로 해당 파일을 참고하는 식으로 관리했습니다.

기존 개발 방식과의 비교

기존 .NET 구조에서는 프론트와 백을 별도로 관리했는데, Next.js는 하나의 프로젝트 안에서 화면과 서버 로직을 함께 다룰 수 있어 AI가 전체 코드를 한 번에 파악하고 작업하는 데 훨씬 유리했습니다. 배포도 GitHub에 코드를 올리면 Vercel이 자동으로 감지해서 배포해주는 CI/CD 파이프라인 덕분에 수동으로 파일을 서버에 올리던 방식과 비교해 생산성 차이가 크게 느껴졌습니다.

문제 상황

목록 조회 시 각 행마다 연관 테이블을 반복 조회하는 Correlated subquery 구조로 인해 1,000건 기준 응답시간이 1,805ms 까지 증가. 데이터 수가 늘수록 내부 실행 횟수도 선형으로 증가하는 구조적 문제

사용 도구
  • P6Spy : JDBC 드라이버 레벨에서 실제 SQL 실행시간을 콘솔에 실시간 출력. 의존성 추가만으로 적용 가능
해결 방법

연관 테이블을 GROUP BY로 미리 집계 후 LEFT JOIN으로 한 번에 연결. 행마다 반복되던 서브쿼리를 단일 집계로 대체. 쿼리 실행 횟수를 2,000회에서 1회로 줄여 응답시간을 90% 이상 단축

코드 비교
BEFORE - 1,805ms (행마다 서브쿼리 2회)
SELECT P.PEN_ID, P.PEN_NAME,
  (SELECT MIN(RM_PRICE)
   FROM PENSION_ROOM
   WHERE PEN_ID = P.PEN_ID -- 행마다!
  ) AS RM_PRICE,
  (SELECT MAX(RM_PRICE)
   FROM PENSION_ROOM
   WHERE PEN_ID = P.PEN_ID -- 행마다!
  ) AS RM_PRICE_MAX
FROM PENSION P
LEFT JOIN PENSION_ATTACH A
  ON P.PEN_ID = A.PEN_ID
LEFT JOIN REVIEW R
  ON A.PEN_ID = R.PEN_ID
GROUP BY P.PEN_ID, P.PEN_NAME
LIMIT 0, 1000
-- 1,000행 x 2회 = 내부 2,000회 실행
-- P6Spy 실측: 1,805ms
AFTER - 90ms (GROUP BY 1회 집계)
SELECT P.PEN_ID, P.PEN_NAME,
  IFNULL(RM_AGG.MIN_PRICE, 0)
    AS RM_PRICE,
  IFNULL(RM_AGG.MAX_PRICE, 0)
    AS RM_PRICE_MAX
FROM PENSION P
LEFT JOIN PENSION_ATTACH A
  ON P.PEN_ID = A.PEN_ID
LEFT JOIN REVIEW R
  ON A.PEN_ID = R.PEN_ID
LEFT JOIN (         -- 1회만 실행
  SELECT PEN_ID,
    MIN(RM_PRICE) AS MIN_PRICE,
    MAX(RM_PRICE) AS MAX_PRICE
  FROM PENSION_ROOM
  GROUP BY PEN_ID
) RM_AGG ON P.PEN_ID = RM_AGG.PEN_ID
LIMIT 0, 1000
-- P6Spy 실측: ~90ms (95%+ 개선)

문제 상황

검색 결과를 static HashMap에 캐싱. 삭제 로직 없어 검색어 종류만큼 힙에 무한 누적 → OOM 위험. 서비스 운영 중 힙 사용량이 지속 증가하며 장시간 운영 시 서버 재시작이 필요한 상황

사용 도구
  • Redis : 별도 메모리 서버에 캐시 저장
  • TTL 10분 : 자동 만료로 힙 사용 없음
  • @Cacheable : 어노테이션 한 줄로 캐시 적용
해결 방법

@Cacheable 애노테이션 추가. 동일 검색어 재요청 시 DB 미조회, Redis 에서 즉시 반환. 캐시 저장소를 JVM 힙에서 Redis 로 분리해 애플리케이션 메모리 부담을 완전히 제거

BEFORE - static HashMap (메모리 누수)
private static final Map<String,
    List<PensionDTO>> CACHE
    = new HashMap<>();

public List<PensionDTO>
getPensionList(Map map) {
    String key = map.toString();
    if (CACHE.containsKey(key))
        return CACHE.get(key);

    List result =
        dao.getPensionList(map);
    // 저장만 하고 삭제 안 함
    // → 힙 무한 증가 → OOM
    CACHE.put(key, result);
    return result;
}
AFTER  - @Cacheable (힙 사용 없음, TTL 자동)
@Cacheable(
    value = "pensionSearch",
    key = "#map.toString()"
)
public List<PensionDTO>
getPensionList(Map map) {
    // HIT: 메서드 자체가 실행 안 됨
    //   → Redis에서 즉시 반환
    // MISS: DB 조회 후
    //   → Redis에 자동 저장 (TTL 10분)
    // Redis 장애 시 → DB 폴백 자동
    return dao.getPensionList(map);
}
// 힙 사용: 0  /  OOM 위험: 제거
느린 검색 API → Redis 캐싱 개선
매 요청마다 DB 조회 (287ms) → @Cacheable + @CacheEvict 로 3ms 달성
P6Spy 실측  |  Redis 3.0  |  @Cacheable + @CacheEvict
문제 상황

동일 검색어 반복 요청에도 매번 DB 쿼리 실행. 평균 287ms, 트래픽 증가 시 과부하. 신규 데이터 등록 시 캐시 불일치 문제. 동시 사용자 증가 시 동일 쿼리가 중복 실행되어 DB 커넥션 풀 고갈 위험도 존재

사용 도구
  • P6Spy : SQL 실행 여부 실측
  • @Cacheable : 캐시 HIT 시 DB 미조회
  • @CacheEvict : 데이터 변경 시 캐시 자동 삭제
해결 방법

@Cacheable로 검색 결과 Redis 캐싱. 신규 등록 시 @CacheEvict로 캐시 즉시 삭제 → 항상 최신 데이터 보장. TTL 10분 설정으로 오래된 캐시는 자동 만료되어 별도 관리 없이도 데이터 정합성 유지
